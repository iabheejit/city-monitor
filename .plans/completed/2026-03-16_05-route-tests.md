# Plan: Route Tests for pollen, feuerwehr, council-meetings, noise-sensors

**Type:** feature (test coverage)
**Complexity:** simple
**Date:** 2026-03-16

## Context

Four route files lack test coverage: `pollen.ts`, `feuerwehr.ts`, `council-meetings.ts`, `noise-sensors.ts`. All four follow the identical cache-hit / DB-fallback / 404 / empty-response pattern used by every other route. The existing test files (e.g., `weather.test.ts`, `wastewater.test.ts`, `population.test.ts`) use the same boilerplate: spin up the app with `skipScheduler: true`, listen on port 0, and test via `fetch`.

Note: existing tests do NOT test the DB fallback path (they only test cache hit, empty/null, and 404). The task description mentions "DB fallback" but the established pattern does not mock the DB layer. We will match the existing pattern exactly (3 tests per route: empty, cache hit, 404) rather than introducing a new DB-mocking pattern, which would be inconsistent and out of scope.

## Files to Create (4)
